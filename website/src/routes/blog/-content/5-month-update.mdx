    date="2026-02-20"
    title={<a {...trackedLinkProps("https://sinclairzx81.github.io/typebox/")}>TypeBox</a>}
    icon={<MdSymbol style={{ color: "var(--warning)" }}>deployed_code</MdSymbol>}
  >

    On initial release of our benchmarks, TypeBox was consistently underperforming in runtime performance.

    After discussion with `sinclairzx81` (author of TypeBox), it emerged that the `v1.x` release had introduced a change in behaviour, specifically that schemas would try to coerce invalid values into valid ones. Understandably, this made the library slower than most other libraries that don't do this.

    [A `v1.1.0` release](https://github.com/open-circle/schema-benchmarks/issues/55#issuecomment-3937075290) made this behaviour opt-in, bringing TypeBox's performance back in line with other libraries.

    ```ts
    // v1.0.x
    // A = 12345
    const A = Value.Parse(Type.Number(), "12345");

    // v1.1.x
    // throw! - Expected Number
    const A = Value.Parse(Type.Number(), "12345");
    ```

  <TimelineItem
    date="2026-03-05"
    title={
      <Link to="/codec">
        Codecs
      </Link>
    }
    icon={<MdSymbol style={{ color: "var(--info)" }}>speed</MdSymbol>}
    >
    Measuring the runtime performance of "codecs", two-way schemas that support both encoding and decoding of data.

    ```ts
    import * as z from "zod";

    const bigIntFromString = z.codec(z.string(), z.bigint(), {
      encode: (bigInt) => bigInt.toString(),
      decode: (str) => BigInt(str),
    });

    bigIntFromString.encode(123n); // "123"
    bigIntFromString.decode("123"); // 123n
    ```

  </TimelineItem>

  <TimelineItem
    date="2026-03-05"
    title={
      <a {...trackedLinkProps("https://decoders.cc")}>
        Decoders
      </a>
    }
    icon={<MdSymbol style={{ color: "var(--warning)" }}>deployed_code</MdSymbol>}
    >

    Based on our <Link to="/download">download</Link> benchmarking, we were able to identify a tree shaking issue with `decoders`, and [collaborated](https://github.com/nvie/decoders/pull/1290) with the author to get it fixed.

    Despite already having `"sideEffects": false`, we found that manually annotating functions with `#__NO_SIDE_EFFECTS__` and function calls with `#__PURE__` allowed bundlers to tree shake the library more effectively, reducing the overall bundle size for users.

    ```ts
    // #__NO_SIDE_EFFECTS__
    function exampleFunction() {
      // function implementation
    }

    const result = /* #__PURE__ */ anotherFunction();
    ```

  </TimelineItem>

Schema Benchmarks now has a wide range of benchmarks (8 different types), and measures the vast majority of notable schema validation libraries (currently <Link to="/libraries">19</Link>). It has helped library authors identify performance regressions,
and provides a useful resource for developers to compare libraries.

If you have any suggestions for new benchmarks, or would like to see a library added, we'd love to hear from you! Please open an [issue](https://github.com/open-circle/schema-benchmarks/issues) on GitHub.
